## 车险理赔预测系统 — 业务流程详细描述
### 一、系统总体架构与角色定位
本系统采用 前后端分离的三层分布式架构 ，面向保险公司内部管理人员（管理员/普通用户），以 车险保单数据管理 为核心业务主体，以 深度学习理赔预测 为智能辅助功能。系统的整体业务流程围绕" 数据录入 → 数据管理 → 智能预测 → 结果分析 "这一主线展开，各功能模块之间通过 RESTful API 实现松耦合的数据交互。

### 二、用户登录与身份认证流程
系统启动后，用户首先通过浏览器访问前端应用入口。前端页面加载时向 Spring Boot 后端发起登录请求，后端接收用户提交的工号（employee_no）与密码（password），在 sys_user 表中进行身份验证（密码采用 MD5 加密存储比对）。验证通过后，后端返回用户基本信息（姓名、角色等）及认证状态，前端据此完成会话建立并跳转至系统主界面。系统预设了默认管理员账户（工号：000001），支持 admin 与 user 两种角色权限的区分管理。该环节确保了后续所有业务操作均在合法的身份上下文中执行。

### 三、主页面数据概览流程
用户成功登录后进入系统首页（Home 页面），该页面作为系统的全局信息汇总中心，自动调用后端 /motorInsurance/overallStatistics 接口从 motor_insurance 表中聚合计算四项核心统计指标——总保费收入、总索赔成本、本年度索赔次数以及平均索赔率，并以卡片形式直观展示于页面顶部。此流程为管理人员提供了业务全局态势的一目了然视图，帮助其快速把握当前保单规模与理赔风险的整体状况。

### 四、基础数据管理业务流程 
4.1 保单信息管理
保单信息是整个系统最核心的基础业务实体，存储于 motor_insurance 表中，包含合同日期、被保人信息、车辆属性、保费金额、历史索赔记录等 28 个维度字段。用户进入「保单信息」页面后，系统执行以下标准 CRUD 业务流程：

- 查询浏览 ：前端发送分页查询请求至后端 /motorInsurance/selectPage 接口，后端经 Controller → Service → MyBatis Mapper 链路从数据库中按页获取保单列表，支持按风险类型（Type_risk）和地区（Area）进行条件筛选，最终以表格形式展示保单 ID、净保费、历史索赔次数等关键字段。
- 新增保单 ：用户点击新增按钮弹出表单，填写完整的保单字段后提交，数据经 JSON 格式传递至 /motorInsurance/add 接口写入数据库。
- 编辑保单 ：用户选中某条保单记录并修改字段值后，调用 /motorInsurance/update 接口完成数据更新。
- 删除保单 ：用户确认删除操作后，调用 /motorInsurance/delete/{id} 接口移除对应记录。 
4.2 车辆信息管理
用户进入「车辆信息管理」页面，对 motor_insurance 表中与车辆相关的属性子集（注册年份、马力、排量、车辆价值、车门数、燃料类型、车长、车重等）进行独立的查看与管理操作，为后续预测模型提供准确的车辆特征输入。
 4.3 理赔类型管理
用户进入「索赔类型」页面，对 claim_types 表中的理赔分类数据进行维护，包括查看各类型的当年索赔成本（Cost_claims_year）和按类型划分的索赔成本（Cost_claims_by_type）。该模块支撑了理赔统计维度的数据基础。

### 五、统计分析可视化流程
系统提供三个维度的统计分析页面，均基于 ECharts 图表库实现数据的可视化呈现：

- 保单统计 （MotorInsuranceStatistics）：调用后端聚合接口，以柱状图、饼图等形式展示保单数据的分布特征（如各地区保单数量占比、各风险类型保费分布等）。
- 理赔统计 （ClaimStatistics）：基于 claim_types 与 motor_insurance 表关联数据，展示理赔成本构成、索赔频率分布等分析图表。
- 预测统计 （PredictionStatistics）：基于 insur_pred 表中的历史预测结果，以图表形式呈现风险等级分布（LOW/MEDIUM/HIGH）、理赔概率直方图、预计理赔金额区间分布等关键指标。
上述统计分析流程的本质是从原始业务数据中提取有价值的信息洞察，辅助管理层做出数据驱动的决策判断。

### 六、核心业务流程 —— 理赔率预测
这是本系统最具特色的核心业务流程，涉及前端、Spring Boot 后端与 FastAPI ML 服务三方的协同工作，完整链路如下：

步骤一：保单选择

用户进入「预测管理」页面，页面首先加载全部可选保单列表（来源于 motor_insurance 表）。用户通过表格的多选框勾选 1 至 10 条待预测的保单记录（系统强制限制单次批量上限为 10 条，防止过载），页面实时显示已选数量。

步骤二：发起预测请求

用户点击「开始预测」按钮后，前端将已选保单 ID 列表以 POST 请求方式发送至 Spring Boot 后端的 /insurPred/predict 接口。

步骤三：后端业务编排

后端 InsurPredService 接收请求后依次执行以下逻辑：

1. 参数校验 ：验证 ID 列表非空、数量不超过 10 条、每项非空；
2. 数据查询 ：根据 ID 列表从 motor_insurance 表中查出对应的完整保单记录；
3. 完整性校验 ：比对请求 ID 与实际查到的保单 ID，发现不存在的则抛出异常提示；
4. 远程调用 FastAPI ：将保单记录组装为 JSON 数组，通过 Hutool HTTP 工具向 FastAPI 服务的 /predict/batch 端点发送批量预测请求。
步骤四：ML 模型推理

FastAPI 服务收到批量预测请求后：

1. 通过 Pydantic 模型校验请求数据格式与字段完整性；
2. 调用 InferenceService 加载预训练好的 MLP 双任务神经网络模型（ best_model.pth ）和标准化器（ scaler.pkl ）；
3. 对每条保单记录进行 特征预处理 （字段映射、缺失值处理、标准化归一化）；
4. 将预处理后的特征向量送入 MLP 模型前向传播，同时输出两个任务头的结果：
   - 分类头输出 ：理赔概率 logit → 经 Sigmoid 映射为 0~1 的概率值，再与最优阈值比较生成 0/1 理赔标签及 LOW/MEDIUM/HIGH 三级风险等级；
   - 回归头输出 ：log1p(理赔金额) 预测值 → 经 expm1 还原为原始尺度的预计理赔金额；
5. 汇总所有记录的预测结果，附加模型版本号与生成时间戳，返回结构化的 JSON 响应。
步骤五：结果持久化与返回

Spring Boot 后端接收到 FastAPI 返回的预测结果后：

1. 逐条解析预测数据，将其转换为 InsurPred 实体对象；
2. 通过 MyBatis Mapper 将每条预测记录插入 insur_pred 表中（含理赔概率、风险等级、预计理赔金额、使用阈值、模型版本、预测时间等字段）；
3. 将入库后的完整记录列表返回给前端。
步骤六：前端展示

前端接收预测响应后，刷新页面下方的「历史预测记录」表格，以表格形式展示每条记录的预测 ID、关联保单 ID、理赔概率（百分比）、风险等级（彩色标签）、预计理赔金额（货币格式）等信息。用户可在此页面继续选择新的保单进行预测，或删除不需要的历史预测记录。
